# M2.5 — HUD de tiempo

Fecha: 2026-05-24

## Objetivo

Mostrar fecha simulada y controles de pausa/velocidad en el frontend, conectados al loop real de `agent-service` via `gateway` y NATS.

## Cambios hechos

### Gateway

- Se agregaron contratos de tiempo en `services/gateway/internal/domain`.
- El WebSocket ahora publica:
  - `tiempo.sesion_iniciada` cuando una conexion validada entra a una partida.
  - `tiempo.sesion_terminada` cuando se cierra la ultima conexion activa de esa partida.
- Se agrego `POST /api/v1/games/{gameID}/time-control`.
- El endpoint valida ownership de la partida antes de publicar eventos.
- El endpoint publica:
  - `tiempo.velocidad_cambiada`
  - `tiempo.pausa_activada`
- El gateway escucha `tiempo.dia_avanzado` y reenvia `time.patch` al frontend.
- El hub cuenta conexiones activas por partida para que dos clientes mirando la misma partida no apaguen el loop antes de tiempo.

### Frontend

- Se agrego `TimeClientState`.
- `useNewGameFlow` aplica deltas `time.patch`.
- `CeremonyPage` muestra:
  - fecha simulada
  - estado de pausa o velocidad activa
  - controles Pausa / x1 / x5 / x20
- Los controles envian cambios al gateway y actualizan el HUD con el delta aceptado.

## Decision tomada

M2.5 no agrega todavia un snapshot REST completo de tiempo.

El HUD arranca con el default canonico `2026-10-01`, pausa y x1, y se sincroniza mediante deltas `time.patch`. Un snapshot de rehidratacion de tiempo queda como mejora natural cuando el frontend de M2 empiece a tener mas vistas persistentes.
